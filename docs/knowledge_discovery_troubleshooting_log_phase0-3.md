# Error 1 - Rust Installed but `rustc` Was Not Recognized

## Phase

Phase 0 - Environment Setup

## Symptom

After installing Rust, PowerShell or VS Code showed errors like:

```powershell
rustc : The term 'rustc' is not recognized
cargo : The term 'cargo' is not recognized
```

## Root Cause

Rust was installed, but the Cargo binary folder was not available in the terminal PATH.

Rust normally installs tools here:

```text
C:\Users\<username>\.cargo\bin
```

That folder contains:

```text
rustc.exe
cargo.exe
rustup.exe
```

If this folder is not in PATH, Windows cannot find the Rust tools.

## Diagnosis Commands

Check whether Rust binaries exist:

```powershell
dir $env:USERPROFILE\.cargo\bin
```

Test Rust directly with the full path:

```powershell
& "$env:USERPROFILE\.cargo\bin\rustc.exe" --version
& "$env:USERPROFILE\.cargo\bin\cargo.exe" --version
```

If those commands work, Rust is installed correctly and the issue is definitely PATH.

## Resolution

Add Cargo's bin folder to the user PATH:

```powershell
$CargoPath = "$env:USERPROFILE\.cargo\bin"

$UserPath = [Environment]::GetEnvironmentVariable("Path", "User")

if ($UserPath -notlike "*$CargoPath*") {
    [Environment]::SetEnvironmentVariable(
        "Path",
        "$UserPath;$CargoPath",
        "User"
    )
    Write-Host "Added Cargo bin to user PATH:"
    Write-Host $CargoPath
} else {
    Write-Host "Cargo bin is already in user PATH."
}
```

Then fully close PowerShell and open a new one.

## Verification

```powershell
rustc --version
cargo --version
rustup --version
```

## Final Status

Resolved.

## YouTube Tutorial Note

This is a good moment to explain that installing Rust is not enough. The terminal must also know where Rust's executables are located. This is one of the most common Windows setup problems for new Rust/Tauri developers.

# Error 2 - Rust Worked in PowerShell but Not in VS Code

## Phase

Phase 0 - Environment Setup

## Symptom

Rust commands worked in normal PowerShell, but failed inside the VS Code integrated terminal.

Example:

```powershell
rustc --version
```

worked in PowerShell but not inside VS Code.

## Root Cause

VS Code was opened before the PATH update. GUI apps on Windows inherit environment variables when they launch. If VS Code was already open before Rust was added to PATH, the integrated terminal still had the old PATH.

## Resolution

Close all VS Code windows.

Open PowerShell and launch VS Code from the project folder:

```powershell
cd C:\Users\van09\Documents\Desktop\KnowledgeDiscover
code .
```

Then test inside the VS Code terminal:

```powershell
rustc --version
cargo --version
```

## Additional Fix

If VS Code still does not refresh:

```text
Ctrl + Shift + P
Developer: Reload Window
```

If that still fails, restart Windows.

## Final Status

Resolved.

## YouTube Tutorial Note

This is worth pointing out because it looks strange at first: Rust works in PowerShell, but not VS Code. The fix is not Rust code. It is environment inheritance. Launching VS Code from the working PowerShell session gives VS Code the correct PATH.

# Error 3 - `sqlite3 --version` Was Not Recognized

## Phase

Phase 0 - Environment Setup

## Symptom

Running:

```powershell
sqlite3 --version
```

failed because the command was not recognized.

## Root Cause

**DB Browser for SQLite** was installed, but the SQLite command-line tool was not installed or not added to PATH.

DB Browser for SQLite is a GUI app. It does not automatically provide the `sqlite3` CLI command.

## Environment State

```text
sqlite3 CLI: not recognized
DB Browser for SQLite: installed
```

## Resolution Used

This was accepted as non-blocking.

Reason:

```text
Phase 1 does not require SQLite.
Phase 2 does not require SQLite.
Phase 3 does not require SQLite.
Phase 4 will create the SQLite App DB through Rust/Tauri code.
DB Browser can inspect the database visually later.
```

## Optional Future Resolution

Install SQLite CLI separately from:

```text
https://www.sqlite.org/download.html
```

Then add the folder containing `sqlite3.exe` to PATH.

For example, if `sqlite3.exe` is located in:

```text
C:\Tools\sqlite
```

Add it to PATH:

```powershell
$SQLitePath = "C:\Tools\sqlite"

$UserPath = [Environment]::GetEnvironmentVariable("Path", "User")

if ($UserPath -notlike "*$SQLitePath*") {
    [Environment]::SetEnvironmentVariable(
        "Path",
        "$UserPath;$SQLitePath",
        "User"
    )
    Write-Host "Added SQLite to user PATH:"
    Write-Host $SQLitePath
}
```

Then reopen PowerShell and verify:

```powershell
sqlite3 --version
```

## Final Status

Deferred. Not blocking.

## YouTube Tutorial Note

This is a good teaching point: not every missing tool blocks the current phase. SQLite CLI is useful, but the project can continue because SQLite is not needed until Phase 4.

# Error 4 - Microsoft C++ Build Tools Requirement

## Phase

Phase 0 / Phase 1

## Symptom

Tauri/Rust builds on Windows require MSVC tools. If missing, builds may fail with compiler or linker errors.

Possible symptoms include:

```text
link.exe not found
cl.exe not found
Microsoft C++ Build Tools missing
```

## Root Cause

Rust desktop builds on Windows often need the MSVC C++ toolchain. Tauri also depends on native Windows compilation tools.

## Required Components

Install through **Visual Studio Build Tools**:

```text
Desktop development with C++
MSVC v143 or latest MSVC build tools
Windows 10 SDK or Windows 11 SDK
C++ CMake tools for Windows
```

## Verification

```powershell
where.exe cl
```

If `cl.exe` is not found after installation, restart Windows.

## Final Status

Resolved.

## YouTube Tutorial Note

This is important to mention in the setup video. Tauri is not just a frontend app. It builds a native desktop application, so Windows native build tools matter.

# Error 5 - Tauri Build Failed Because `icons/icon.ico` Was Missing

## Phase

Phase 1 - Tauri React TypeScript App

## Symptom

Tauri failed with:

```text
`icons/icon.ico` not found; required for generating a Windows Resource file during tauri-build
```

## Root Cause

The Tauri config expected this file:

```text
src-tauri/icons/icon.ico
```

but the icon files had not been generated.

## Resolution

Create a temporary `app-icon.png`, then generate the Tauri icon set.

PowerShell command used to create a simple temporary icon:

```powershell
cd C:\Users\van09\Documents\Desktop\KnowledgeDiscover

Add-Type -AssemblyName System.Drawing

$bitmap = New-Object System.Drawing.Bitmap 512, 512
$graphics = [System.Drawing.Graphics]::FromImage($bitmap)

$background = [System.Drawing.Brushes]::DarkSlateBlue
$textBrush = [System.Drawing.Brushes]::White
$font = New-Object System.Drawing.Font("Arial", 120, [System.Drawing.FontStyle]::Bold)

$graphics.FillRectangle($background, 0, 0, 512, 512)
$graphics.DrawString("KD", $font, $textBrush, 130, 175)

$bitmap.Save("app-icon.png", [System.Drawing.Imaging.ImageFormat]::Png)

$graphics.Dispose()
$bitmap.Dispose()
```

Then generate icons:

```powershell
pnpm tauri icon app-icon.png
```

## Verification

```powershell
dir src-tauri\icons
```

Expected generated files include:

```text
icon.ico
32x32.png
128x128.png
128x128@2x.png
icon.icns
```

Then run:

```powershell
pnpm tauri dev
```

## Final Status

Resolved.

## YouTube Tutorial Note

This is a useful real-world error to show because it proves Tauri needs native app assets. Even in development, Windows may require `icon.ico` for the generated resource file.

# Error 7 - Port 1420 Already in Use

## Phase

Phase 1 / Phase 2 Testing

## Symptom

Running:

```powershell
pnpm tauri dev
```

failed with:

```text
Error: Port 1420 is already in use
Error The "beforeDevCommand" terminated with a non-zero status code.
```

## Root Cause

A previous Vite/Tauri development server was still running on port `1420`.

## Resolution Option 1 - Stop the Existing Terminal

Find the old terminal running:

```powershell
pnpm tauri dev
```

Stop it with:

```text
Ctrl + C
```

Then rerun:

```powershell
pnpm tauri dev
```

## Resolution Option 2 - Kill the Process Using Port 1420

Find the process:

```powershell
netstat -ano | findstr :1420
```

The last number is the process ID.

Kill it:

```powershell
taskkill /PID <PID> /F
```

Replace `<PID>` with the actual process ID.

## Resolution Option 3 - PowerShell One-Liner

```powershell
$pid1420 = (Get-NetTCPConnection -LocalPort 1420 -ErrorAction SilentlyContinue).OwningProcess

if ($pid1420) {
    Stop-Process -Id $pid1420 -Force
    Write-Host "Killed process using port 1420: $pid1420"
} else {
    Write-Host "No process found on port 1420"
}
```

Then rerun:

```powershell
pnpm tauri dev
```

## Final Status

Resolved.

## YouTube Tutorial Note

This is an excellent beginner-friendly debugging point. The code may be fine, but the dev server cannot start because the port is already occupied.

# Error 8 - Phase 2 Commit Used the Wrong Commit Message

## Phase

Phase 2 - Tailwind Desktop Shell

## Symptom

Phase 2 was accidentally committed with the Phase 1 message:

```text
Phase 1: scaffold Tauri React desktop app
```

## Root Cause

Wrong commit message was used during `git commit`.

## Diagnosis

```powershell
git log --oneline -3
```

Showed:

```text
417c83c (HEAD -> main) Phase 1: scaffold Tauri React desktop app
cd285d5 (origin/main) update gitignore
2efb554 Phase 1: scaffold Tauri React desktop app
```

## Resolution

Because the wrong commit had not been pushed yet, amend the latest commit message:

```powershell
git commit --amend -m "Phase 2: add Tailwind desktop app shell"
```

Verify:

```powershell
git log --oneline -3
git status
```

## Push

Because the wrong commit had not been pushed yet, a normal push was enough:

```powershell
git push
```

No force push was needed.

## Important Rule

If the wrong commit had already been pushed, then after amend the correct command would be:

```powershell
git push --force-with-lease
```

But since it had not been pushed, this was not needed.

## Final Status

Resolved.

## YouTube Tutorial Note

This is worth showing because Git mistakes happen constantly. The key rule is: if the bad commit has not been pushed, `git commit --amend` is safe and clean.

# Error 9 - Windows Application Control Blocked `tauri-plugin-dialog`

## Phase

Phase 3 - Vault Creation

## Symptom

Build failed with:

```text
error: failed to run custom build command for `tauri-plugin-dialog v2.7.1`

Caused by:
  could not execute process `...\src-tauri\target\debug\build\tauri-plugin-dialog-...\build-script-build` (never executed)

Caused by:
  An Application Control policy has blocked this file. (os error 4551)
```

## Root Cause

Windows Application Control, Smart App Control, Defender, or another security policy blocked a Rust-generated build script executable created under:

```text
src-tauri\target\debug\build\
```

The blocked dependency was:

```text
tauri-plugin-dialog
```

This was not caused by frontend code or the vault creation logic. It was a local Windows security policy blocking a generated build executable.

## Initial Attempted Fixes

### Attempt 1 - Move Project Away From Desktop

Moved project from:

```text
C:\Users\van09\Documents\Desktop\KnowledgeDiscover
```

to:

```text
C:\Dev\KnowledgeDiscover
```

Reason: Desktop folders can sometimes be monitored more aggressively.

### Attempt 2 - Clean Rust Build Cache

```powershell
Remove-Item -Recurse -Force .\src-tauri\target -ErrorAction SilentlyContinue
```

### Attempt 3 - Reinstall Dependencies

```powershell
pnpm install
```

## Result of Attempted Fixes

The Windows policy still blocked the generated build script.

## Recommended Resolution Option A - Allow the Blocked File

Open:

```text
Windows Security
→ Virus & threat protection
→ Protection history
```

Look for a recent blocked item related to:

```text
tauri-plugin-dialog
build-script-build
C:\Dev\KnowledgeDiscover\src-tauri\target\debug\build\
```

If available, choose:

```text
Allow on device
```

Then rerun:

```powershell
cd C:\Dev\KnowledgeDiscover
Remove-Item -Recurse -Force .\src-tauri\target -ErrorAction SilentlyContinue
pnpm tauri dev
```

## Recommended Resolution Option B - Use Manual Path Input Instead of Dialog Plugin

Since the dialog plugin is the dependency being blocked, remove it for now and complete Phase 3 with a manual parent-folder path input.

This keeps the core architecture intact:

```text
React UI
    ↓
Tauri invoke
    ↓
Rust command
    ↓
File system creates KnowledgeDiscoveryVault
```

## Workaround Requirements

Remove frontend dependency:

```text
@tauri-apps/plugin-dialog
```

Remove Rust dependency:

```text
tauri-plugin-dialog
```

Remove dialog plugin registration from:

```text
src-tauri/src/main.rs
```

Replace folder picker UI with manual input:

```text
Parent folder path
Example: C:\Users\van09\Documents\Desktop
```

The app should still call:

```rust
create_vault(parent_path: String) -> Result<VaultInfo, String>
```

## Final Status

Active issue. Workaround planned.

## YouTube Tutorial Note

This is an important tutorial lesson: do not let a convenience feature block the core system. The folder picker is convenient, but the important architectural milestone is React calling Rust to create the vault structure.

# Error 11 - Stale or Incorrect SQLite PATH Entry

## Phase

Phase 0 - Environment Setup

## Symptom

A SQLite path appeared to exist in PATH:

```text
C:\Tools\sqlite
```

but the folder did not exist.

## Root Cause

A stale or incorrect PATH entry was added. The PATH pointed to a folder that did not actually contain `sqlite3.exe`.

## Diagnosis

Check PATH entries containing SQLite:

```powershell
$env:Path -split ';' | Select-String -Pattern "sqlite"
```

Check user PATH entries:

```powershell
[Environment]::GetEnvironmentVariable("Path", "User") -split ';' | Select-String -Pattern "sqlite"
```

Check whether the folder exists:

```powershell
dir C:\Tools\sqlite
```

## Resolution

Remove the bad PATH entry:

```powershell
$BadPath = "C:\Tools\sqlite"

$CleanPath = (
    [Environment]::GetEnvironmentVariable("Path", "User") -split ';' |
    Where-Object { $_ -and $_ -ne $BadPath }
) -join ';'

[Environment]::SetEnvironmentVariable("Path", $CleanPath, "User")

Write-Host "Removed bad SQLite path:"
Write-Host $BadPath
```

Then close and reopen PowerShell.

## Final Status

Not critical. SQLite CLI remains deferred because DB Browser for SQLite is installed.

## YouTube Tutorial Note

This is a good example of PATH troubleshooting. A command can fail even when PATH appears to contain a related folder if the folder is wrong or missing.

# Lessons Learned So Far

## 1. Windows PATH Issues Are Common

Rust, SQLite, and VS Code all had environment/PATH-related issues. The reliable pattern is:

```text
Install tool
Verify executable exists
Add correct folder to PATH
Close and reopen terminal
Launch VS Code from PowerShell if needed
```

## 2. VS Code Can Have a Stale Environment

If VS Code was open before PATH changed, its integrated terminal may not see new tools.

Best workaround:

```powershell
code .
```

from a PowerShell terminal that already sees the tools.

## 3. Tauri Requires Native Windows Build Assets

Missing icons can break the Tauri build even during development.

Required file:

```text
src-tauri/icons/icon.ico
```

## 4. Not Every Missing Tool Blocks the Current Phase

SQLite CLI was missing, but it did not block Phase 1, Phase 2, or Phase 3.

The correct decision was to defer it.

## 5. Port Conflicts Are Not Code Errors

`Port 1420 is already in use` means an old dev server is still running.

## 6. Git Commit Messages Can Be Fixed

If the latest commit has the wrong message and was not pushed:

```powershell
git commit --amend -m "Correct message"
```

## 7. Security Policy Can Block Rust Build Scripts

Windows Application Control can block generated Rust build executables under:

```text
src-tauri/target/debug/build
```

This is not necessarily bad code.

## 8. Simplify When a Convenience Dependency Blocks Progress

The dialog folder picker is useful, but not essential.

Manual path input lets Phase 3 continue while preserving the important architecture.
